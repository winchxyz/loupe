# Digital design fundamentals — applied rules

Core idea: a digital product wins on experience, not feature parity. The same problem solved with a better flow beats the incumbent (Uber/Lyft vs. flagging a cab). So treat experience as the differentiator, and sequence the work so foundations come before surface.

## Rules (each = a build decision)
1. Compete on experience, not features — when two solutions do the same job, ship the one with the smoother flow. Audit every screen for friction the user shouldn't have to tolerate; remove a tap before adding a feature.
2. Build the fundamentals first; skin them last — grids, color palettes, composition, and icons are portable foundations. Lock the grid system before styling, because a robust grid translates across web, print, and any new layout you bolt on later. Do NOT start from a visual "look" and reverse-engineer structure.
3. Sequence the work in order — identify the audience, do research, weigh ethics and accessibility, THEN design icons / grids / color / composition. Skipping straight to composition is the common mistake; the upstream steps decide what the composition is even for.
4. Treat accessibility and ethics as design inputs, not a final-pass checklist — decide them while choosing palette and layout (contrast, motion, who could be harmed), so they shape the design instead of patching it.
5. Self-pace and treat exercises as milestones — don't cram a build into one weekend. Break the work into checkpoints (audience → research → grid → color → composition) and finish each before moving on; give each concept enough time to actually land.
6. Iterate — designs (and systems) get better through revision, not on the first pass. Plan to revisit and refine; ship a version, learn, improve. Bake in a feedback/iteration step rather than treating v1 as final.
7. Carry concepts across media — the gap between web, print, and industrial design is mostly terminology and tooling. Don't dismiss a technique because it came from another discipline; a print grid idea can serve a web layout and vice-versa.
8. Keep Play in the loop — exploration is a fundamental of design, not a detour. Budget time to try a riskier variant before committing to the safe one.

## Build checklist (use when kicking off a site/UI)
- [ ] Audience identified and at least light research done before any layout?
- [ ] Grid system locked before styling — and is it robust enough to survive a redesign?
- [ ] Accessibility + ethics considered while choosing palette/composition, not after?
- [ ] Build sequenced (audience → research → grid → color → composition), not started at the surface?
- [ ] Does this flow beat the incumbent on experience, or just match it on features?
- [ ] An iteration/feedback pass planned, not a one-shot v1?
- [ ] One exploratory ("Play") variant tried before locking the safe choice?
