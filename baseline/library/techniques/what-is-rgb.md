# RGB color model — applied build rules

Core idea: screens make color by ADDING red, green, and blue light. All three off = black (the screen emits nothing); all three full = white. Every digital color you author is just three intensities, 0–255 per channel. Build with that mental model and the surprises ("why is my hex muddy / why doesn't it print") stop happening.

## Build decisions (each = a do/choose/avoid)
1. Author in RGB for anything on a screen — web, app UI, LED/monitor, photo, video. If light makes the image, it is RGB. Avoid pulling print (CMYK) swatches straight into screen designs; they were mixed for a different physics.
2. Treat 0–255 per channel as your real range. `#000000` = all channels at 0 (true off/black), `#FFFFFF` = all at 255 (true on/white). Reach for exact endpoints deliberately — pure-black backgrounds on OLED literally emit nothing, pure-white maxes brightness/glare.
3. Remember it's ADDITIVE, so brightness stacks. Layering glows, additive blend modes, or stacked translucent light pushes channels toward 255 and clips to white. When a bright effect "blows out," you've saturated a channel — pull intensities down, don't add more layers.
4. Prefer a format that exposes the channels you'll actually tune. Hex is fine for static tokens; use `rgb()` / `rgba()` (or HSL/OKLCH) when you need to programmatically dim, tint, or alpha a color — alpha and per-channel math are clearer there than in hex.
5. Build dark↔light as channel intensity, not as separate ad-hoc colors. A "darker shade" is the same hue at lower channel values; a "lighter tint" is higher values toward white. Generate ramps by scaling intensity so a palette stays coherent.
6. Never assume on-screen color survives to paper. RGB does not map 1:1 to CMYK — saturated screen colors (vivid blues, neon greens) sit outside print gamut and dull on press. If a design also prints (event graphic, poster), convert RGB→CMYK via a color profile or converter and re-check the hues, don't eyeball it.
7. Keep one source-of-truth color profile for screen work (sRGB unless you have a reason). Mixed/unmanaged profiles are why "the same hex" looks different across exports — pin the profile so channels mean the same thing everywhere.

## Build checklist (tick while setting up a palette)
- [ ] Palette authored in a screen-native space (RGB/hex/HSL/OKLCH), not lifted from print swatches?
- [ ] True endpoints used intentionally — `#000`/`#FFF` only where pure off/full-on is wanted?
- [ ] Bright/glow/additive stacks checked for channel clipping to white?
- [ ] Shades & tints generated by scaling channel intensity, so the ramp stays coherent?
- [ ] If it will ALSO print: converted RGB→CMYK and re-verified the hues, not assumed they'd match?
- [ ] One pinned color profile (e.g. sRGB) so a hex means the same color across every export?
