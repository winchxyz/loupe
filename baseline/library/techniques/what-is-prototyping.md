# Prototyping fidelity — applied rules

Core idea: a prototype is a *testable model built to validate before you spend build hours*. The whole game is matching fidelity to the question you're trying to answer — lo-fi to validate the concept cheaply, hi-fi to validate the real interaction. Over-build the wrong stage and you've spent polish on an idea that wasn't proven; under-build the final stage and your user test lies to you.

## The rules (each = a build decision)
1. Prototype to answer a question, not to "look done" — before building, name what this model must validate (the concept? the flow? one feature's interaction?). The question sets the fidelity, not your urge to polish.
2. Validate the concept at low fidelity FIRST — sketches, basic wireframes, simple clickable mockups. Hours to 1-2 days, not weeks. If stakeholders request new functionality, prove or kill the idea here before any real design/dev hours go in.
3. Keep lo-fi lo-fi — simple transitions, flowchart-level structure, just enough detail for a proof of concept. Resist adding color/polish; detail at this stage hides whether the *structure* works.
4. Go high-fidelity only when the question is realism — polished interactions, micro-interactions, conditional logic, real animations. Reach for it for genuine user testing, final client/exec approval, or developer handoff. Days to weeks; spend that cost deliberately.
5. You can scope hi-fi to ONE feature — midway through, build a high-fidelity model of a single high-priority feature while everything else stays rough. Don't gate the whole screen on full polish.
6. Pick fidelity by what you're testing, per this table:
   - Explore many concepts fast / early stakeholder buy-in / basic flow + IA → **low fidelity**.
   - Realistic user testing / final approval / dev handoff spec → **high fidelity**.
7. A hi-fi prototype can become the product — build hi-fi interactions as real, reusable components (conditional logic, micro-interactions, polished animation) so the validated prototype hands off cleanly instead of being thrown away.
8. Match the prototype type to the challenge: paper = cheapest to change but poor at conveying the real feel; lo-fi digital = engaging interactive flow without heavy build; hi-fi digital = color, buttons, menus, real interactive elements for true-to-life feedback.

## Build checklist
- [ ] Named the ONE question this prototype must answer before building it?
- [ ] Fidelity matched to that question (lo-fi to validate concept, hi-fi to validate interaction)?
- [ ] Concept validated at low fidelity before any expensive polish went in?
- [ ] Hi-fi scoped to the high-priority feature(s), not blocking the whole view on polish?
- [ ] Hi-fi interactions built as reusable components so a validated prototype survives handoff?
- [ ] Got real feedback on the model before committing build hours to the full design?
