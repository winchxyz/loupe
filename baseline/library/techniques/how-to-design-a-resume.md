# Content-first information architecture — applied rules

Core idea: a résumé is the smallest possible "sell yourself on one page" problem, so its rules generalize to any bio/about/profile/summary surface. The page is an advertisement for a person; the reader scans, ranks, and decides in seconds. Lock the CONTENT and its priority order BEFORE touching typeface or layout — visual choices then express a hierarchy that already exists, instead of inventing one.

## The rules (each = a build decision)
1. Write the content first — choose type/layout last. Get every fact down (name, contact, portfolio link, education, work history, role-relevant skills) before you pick a typeface or grid. If you style before the copy is final, the layout fights the content. Avoid designing an empty shell and pouring text in.
2. Rank by relevance, then map rank → hierarchy. Put the most relevant experience at the top and give it the strongest visual weight (size/weight/space). Hierarchy is not decoration — it is the priority order made visible.
3. Group by Similarity + Proximity, not by borders. Like items become one visual set by sharing style and sitting close together; unlike sets are separated by a bigger gap. Three jobs + two degrees = two clearly distinct clusters formed by spacing and consistent styling, not boxes or rules. Spacing IS the grouping signal.
4. Keep entry structure consistent across a group. Every job/education entry carries the same fields in the same order — title, company, date range, description of work + responsibilities. Consistent sub-structure is what lets Similarity read as "these are the same kind of thing."
5. Express personality through type, not ornament. No personal logo, no invented identity mark. Voice and character come from typeface choice and pairing — restraint here keeps a self-presentation page credible rather than gimmicky.
6. Tailor the content to the target. Research the company/role/team and reorder/trim so the most relevant work rises. The same facts get a different hierarchy for a different audience — relevance is contextual, not absolute.
7. Constrain to one page (one frame). Unless the history is genuinely vast, force everything onto a single page. The hard size limit is the forcing function that makes you cut and rank instead of dumping.
8. Set the canvas to the real output size up front. The exercise frame is A4; switch to US Letter (8.5×11") for US/Canada. Pick the true target dimensions before laying out so spacing and line lengths are real, not approximate — a web analogue of designing at the actual viewport.
9. Get an outside review before calling it done. Hand it to a fresh pair of eyes. You cannot scan your own hierarchy cold; a second reader catches what your familiarity hides.

## Build checklist (use when reviewing a self-presentation page)
- [ ] Content finalized before type/layout was chosen (not the reverse)?
- [ ] Single clear priority order — most relevant item top + heaviest visual weight?
- [ ] Like items grouped by spacing + consistent styling, unlike sets split by a bigger gap (no reliance on boxes)?
- [ ] Every entry in a group shares the same field structure (title / org / date range / description)?
- [ ] Personality carried by typeface choice, not by a logo or ornament?
- [ ] Canvas at the real output size, content fits one page/frame?
- [ ] Reviewed by someone other than you before shipping?

> Ties to our experience: this is the same grouping logic behind our PAIN.md repos-table header misalignment — when entries in a set stop sharing structure (a column drifting out of alignment), Similarity/Continuity break and the cluster stops reading as one coherent group.
