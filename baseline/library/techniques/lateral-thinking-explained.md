# Lateral thinking — applied rules

Core idea: vertical thinking digs the obvious hole deeper (logical, step-by-step, refines one path); lateral thinking jumps sideways to a different starting point. Use vertical thinking to BUILD and validate; use lateral thinking BEFORE you commit, when the first concept is just the conventional one. Each technique below is a concrete move to run when a design feels safe, templated, or stuck.

## The moves (each = a concept decision, not a vibe)
1. Reverse-brainstorm — instead of "how do I make this good," ask "how would I make this maximally bad / annoying / confusing," then invert every answer into a rule. Fastest way to surface the real failure modes of a layout or flow.
2. Bad-ideas pass — deliberately generate the dumbest possible solutions, then mine each: what's the kernel that's actually interesting? The "wrong" idea often hides the non-obvious right one. Do this before you fall in love with idea #1.
3. Random-entry / random-word — pair the problem with an unrelated word and force a connection (de Bono's technique; Figma's example: "video editing should be fun" + "bicycle" → animated map of bike routes). Use to escape a concept rut: pull a random noun, ask "what does THIS suggest for the hero / nav / transition?"
4. SCAMPER the existing thing — run seven prompts on a current design: Substitute, Combine, Adapt, Modify, Put-to-another-use, Eliminate, Reverse. Best when iterating on something that already exists rather than starting cold. "Eliminate" and "Reverse" produce the boldest moves — what if the page had no nav? what if the scroll ran the other way?
5. Six thinking hats — evaluate a concept from six fixed angles instead of one gut take: white=facts, yellow=benefits, black=risks, red=feelings/intuition, green=alternatives, blue=process control. Run the green hat to generate, the black hat to pressure-test, the red hat to check whether it actually FEELS right before committing.
6. Mind-map the problem — start from one central question and branch outward to expose unexpected connections instead of marching down a list. Use for complex multi-section sites where the structure itself is the open question.
7. Provocation / challenge the assumption — name the constraint everyone treats as fixed ("a portfolio needs a grid of thumbnails," "a hero needs a headline") and deliberately violate it to see what opens up.

## When to reach laterally vs. stay vertical
- First concept feels like the default for this product type → go lateral (moves 2, 3, 7).
- Concept chosen, now refining feasibility / polish → stay vertical (logical, step-by-step).
- Iterating on an existing build → SCAMPER (move 4).
- Evaluating between concepts → six hats (move 5).

## Build checklist (run before committing to a direction)
- [ ] Did I challenge the one assumption everyone makes about this product type, or just inherit it?
- [ ] Have I generated at least one deliberately "bad" or random-seeded alternative and mined it?
- [ ] Ran a black-hat pass — what's the risk/failure mode of this concept?
- [ ] Ran a red-hat gut check — does it actually feel distinctive, not just correct?
- [ ] Switched back to vertical thinking to validate feasibility before building?

> Ties to our experience: this is the antidote to "first idea = default layout." 02b only got interesting once we jumped sideways from a normal scroll page to a water→space→sun cinematic journey — and the section-design phase explicitly kept categories OPEN to change if research finds something better, which is lateral thinking as policy.
