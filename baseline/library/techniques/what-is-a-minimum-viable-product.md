# Minimum viable product — applied build decisions

Core idea: ship the smallest version that genuinely solves the core problem and produces real feedback. "Minimal" is about feature count; "viable" is the non-negotiable — a stripped build that doesn't actually work for the user is wasted effort, not an MVP. Build to learn, not to impress.

## The rules (each = a build decision)
1. Define ONE core problem before scoping — write the single hypothesis the build tests. If you can't state it in a sentence, you'll over-build. Everything that doesn't serve it is cut for v1.
2. Pass all four viability gates, not just "minimal": (a) cheap to produce/replicate, (b) the main idea is obvious to the user on sight, (c) it solves a real problem / adds value, (d) it's adaptable — leaves room for later builds. Ship only when all four hold.
3. Protect viability over polish — choose foundational value first, then refinement, then delight. Never trade a working core for a polished detail. A useful-but-not-viable cut is a non-ship.
4. Build to gather feedback, not to reach perfection — choose rapid validation over extensive pre-launch polishing. Resist the urge to finish every edge state before real users touch it.
5. Instrument before you launch — decide up front which metrics prove viability: usage frequency, feature-discovery patterns, engagement. If a build can't be measured, you can't learn from it.
6. Let constraints drive the design — when you must deprioritize, return to the core objective; the forced focus produces sharper, more differentiated solutions than unlimited feature expansion.
7. Run the Build → Measure → Learn loop explicitly: Build a minimal prototype to test a hypothesis; Measure real user behavior with chosen metrics; Learn by answering honestly — did we identify the right problem, did the build solve it, did anything unexpected surface? Each pass feeds the next iteration; repeat continuously.
8. Use a prioritization matrix to pick the cut — plot features by urgency x impact; build the high-urgency/high-impact quadrant for v1, defer low/low. Make the cut line visible and deliberate, not vibes.

## Build checklist (use when scoping any new build)
- [ ] One core problem / hypothesis written in a single sentence?
- [ ] All four viability gates pass (affordable, obvious idea, solves a problem, adaptable)?
- [ ] Foundational value secured before any polish or delight work?
- [ ] Success metrics chosen and instrumented BEFORE launch (usage / discovery / engagement)?
- [ ] Features ranked on an urgency x impact matrix, with a deliberate cut line?
- [ ] A Measure → Learn pass scheduled so feedback actually shapes the next iteration?
