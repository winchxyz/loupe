# Human-centered design — applied rules

Core idea: put the real person at the center of every decision. Start from an actual user problem in its real-world context, build empathy, then design/test/iterate WITH users — not for an imagined one. The point is solving the right problem, not shipping clever features.

## Rules (each = a build decision)
1. Lead with a real user problem — before opening any layout, name the specific problem this view solves and who has it. No problem statement → no build. Resist the instinct to jump straight to a vision or a slick solution.
2. Empathize first, design second — base structure and copy on observed user needs/feelings, not on what looks impressive. Run or read user research (interviews, observation) and let it refine or validate the problem statement.
3. Use the product as a customer would — spend real time walking the actual flow end-to-end before deciding anything. Most pain hides where you assumed it was fine; dogfood every mode/state, don't trust a settled-looking frame.
4. Keep "careabouts" top of mind — track the handful of key customer problems and let them drive every iteration. Each change should ease a known pain point, not add chrome.
5. Solve holistically, find the root cause — the visible problem is usually one piece of a bigger puzzle. Drill to the root cause before designing; a local fix to a symptom is wasted effort.
6. Give the user agency — when automating anything (including AI), keep the human in the driver's seat: show what's happening and let them see/select/override. Control and transparency build trust (LinkedIn's recruiter-AI lets users see and pick the fields AI fills).
7. Validate three questions before committing to an idea — does it actually meet user needs? can we build it with available resources/tech? is it sustainable for the business long-term? An idea must clear all three.
8. Prototype to learn, not to impress — build a fast interactive mock-up specifically to test and validate the concept with real testers, then tweak from their feedback.
9. Test and iterate until it meets expectations — don't ship at "looks done." Keep the loop (test → observe → adjust) running until the prototype genuinely satisfies user expectations, then build for real.
10. Treat the process as a loop, not a line — define → research → ideate → prototype → test feeds back into itself; expect to revisit the problem statement as evidence comes in.

## Build checklist (use when starting or reviewing a build)
- [ ] Can I state the specific user problem this view solves, and who has it?
- [ ] Is the structure driven by observed user needs, not by what looks impressive?
- [ ] Have I actually walked the real flow end-to-end, in every mode/state?
- [ ] Did I trace the visible problem to its root cause, not just patch the symptom?
- [ ] Where something is automated, can the user see it and override it?
- [ ] Does the idea pass all three gates — needs / feasibility / sustainability?
- [ ] Have I tested with a real user (or stand-in) and iterated, rather than stopping at "looks done"?
