# Diagram patterns for collaborative workshops — reusable choices

Core idea: a workshop (and any collaborative thinking session) fights attention erosion. The fix is structure made VISIBLE — a purpose-fit diagram per phase carries the cognitive load so people don't have to. Each diagram below is a reusable pattern; pick by the JOB it does, not by its name.

## The patterns (choose by job)

1. **Lotus diagram (3x3 grid)** — goal in the center cell, eight candidate actions in the surrounding cells. Use when: you need to expand ONE topic into a fixed, bounded set of options before any work starts. The bounded 8 forces prioritization; then "pluck" the cells that aren't relevant or feasible. Reach for it to scope a build's feature set or a page's content blocks.

2. **Timeline / agenda** — once priorities exist, assign each a slot in sequence. Use when: order and time-budget matter and you want a shared artifact people can comment on and flag live. The pattern is "ranked list → time-boxed sequence."

3. **Fishbone / Ishikawa (cause → effect)** — spine from root cause (tail) to end effect (head), ribs for contributing factors. Use when: framing a SYSTEMIC problem so the group sees causes vs. symptoms. Composes with deeper tools: 5 Whys for a single quality issue, customer-journey map for the happy/unhappy path, swimlane for cross-team process. Reach for it when "why is this broken" needs to be visual, not narrated.

4. **Affinity diagram (cluster + group)** — dump many ideas, then group them by relatedness. Use when: divergence is producing volume fast and you need to converge without killing momentum (e.g. breakout groups). The pattern is "many notes → emergent clusters → named themes."

5. **Argument map (pros/cons + signal)** — candidate solutions on one screen, pros and cons attached, plus a voting/polling signal (flags, stickies, widgets). Use when: deciding between explored options and you want traction to be visible at a glance. The pattern is "options + weighted evidence + a read of the room."

6. **SMART goals diagram** — convert a chosen direction into Specific / Measurable / Attainable / Relevant / Time-bound commitments, each assignable to an owner. Use when: closing a session so momentum becomes action. Time-box and distribute sub-tasks so no one is overburdened (e.g. "one slide per presenter per week").

## Cross-cutting build decisions
- Match diagram to PHASE: expand (lotus) → schedule (timeline) → diagnose (fishbone) → converge (affinity) → decide (argument map) → commit (SMART). Don't reuse one diagram for a job it doesn't fit.
- Prefer a diagram over prose when the relationship (sequence, cause, cluster, trade-off) IS the content — it lightens cognitive load and reads faster.
- Front-load the boring stuff: circulate intros/context BEFORE the live session, not during it. On a page, that maps to "give the reader orienting context up front so the main flow stays uninterrupted."
- Make every artifact collaborative/annotatable where possible — shared comments and live voting beat a static handout.

## Build checklist
- [ ] Is each section/phase paired with the diagram whose JOB actually fits it (not a default choice)?
- [ ] Does any relationship that's currently prose deserve to be a diagram (sequence / cause / cluster / trade-off)?
- [ ] Are bounded grids (like the lotus 8) used to force prioritization rather than open-ended sprawl?
- [ ] Does the final phase convert decisions into owned, time-boxed, SMART next steps?
- [ ] Is orienting context delivered up front so the main flow reads uninterrupted?
